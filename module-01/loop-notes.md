We will be using Claude as if it is a junior engineer who needs direction, where we will essentially hand hold them with what exactly we want done. The first step is to plan and this is where we will write a precise spec or task description, you can think of this as a ticket issued on what needs to be done. We will then implement the task by letting Claude write the code; but we don't just trust Claude, we will remain in the loop and approve the tool calls (what Claude uses to complete the task). We then will test the code by running the application or test suite, as we want to ensure what Claude wrote actually works! We then review the diff ourselves, line by line, the exact same way we would review a junior's PR before we merge to catch misunderstandings, scope creep or bugs. Lastly, we will commit only what we have approved manually, which keeps our git history intentional and clean. The review step is the most common failure because Claude is fluent and confident, so code that looks like it works slips through unread, and by letting this happen we have cosigned a change we don't fully own.

I have been using AI assisted coding for a while now and have generally been following this loop, but more so recently. My plan step had been lacking, but has gotten better with creating a spec for the project. My testing has been run the application, but recently been working on both property based testing and unit tests. Lastly, I had tended to skim through the code in the review step and not catch bugs that are noticeable. If I had been using this agent loop more regularly and thoroughly, I would have caught more bugs quicker than before. An example of this is when developing an AWS CDK stack, that the AI would at times give a broader IAM policy than using the concept of Least privilege. AI giving the IAM policy ['dynamodb:*'] and resources: ['*'] instead of ['dynamodb:GetItem'] on resources['arn:aws:dynamodb:us-east-1:123456789012:table/MyTable'] that would follow least privilege, which is best practice.
